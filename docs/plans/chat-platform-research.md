# Chat platform research — Telegram / Slack / Discord / LINE

Status: research, 2026-05-20. Input for `chat-transport-interface.md` (next).

## TL;DR

- **Cleanest fit: Slack via Socket Mode.** It's the closest cousin to Telegram's long-poll model — a WebSocket-pulled event stream, bot token + app-level token, native threads (`thread_ts`), bot-can-add-arbitrary-emoji-reactions, in-channel typing indicator (`assistant.threads.setStatus`). Almost everything in `poller.ts` maps over.
- **Almost as clean: Discord via Gateway WebSocket.** Bot token only, threads inside forum channels each get their own `channel_id` (so "1 project = 1 channel/thread" maps directly), bot reactions are first-class, custom-emoji-friendly. The footgun is the **privileged `MESSAGE_CONTENT` intent** — works free under 100 servers, gated review after.
- **Hardest: LINE.** Webhook-only (no long-poll, no Gateway), **no in-group loading indicator, no per-message reactions, no group threads.** This forces the abstraction to surface "ack channel" and "thread model" as nullable capabilities — leaky abstraction, can't be hidden.
- **Leaky-abstraction summary:** the interface design will need to expose `ackMechanism` (reaction / status / inline-text fallback), `threadModel` (forum-topic-id / sub-channel-id / parent-message-ts / none), and `inboundTransport` (long-poll / websocket / webhook+tunnel) as capability flags. No single normalized "thread id" type works.

## Abstraction-breakers

The interface design must explicitly model these — they cannot be hidden behind a common type.

1. **Identity-unit shape is heterogeneous.** Telegram = `(chat_id, message_thread_id?)`. Slack = `(channel, thread_ts?)`. Discord = `channel_id` only (threads ARE channels). LINE = a tagged union `userId | groupId | roomId`. The "send to here" type must be a discriminated union per platform, not a flat string. The locked F1+F2 invariant ("1 forum topic = 1 project") becomes "**1 addressable thread-equivalent = 1 project**" — and on LINE that's "1 group/DM = 1 project" because no threads exist.

2. **Thread creation model differs.** Telegram and Slack: threads are user-created by the human in the client. Discord: bot can (and often should) create the thread via REST. LINE: no threads. Auto-mount semantics ("user opens new topic → poller auto-creates mount") only work cleanly on Telegram + Discord; Slack threads are dynamic and short-lived (which breaks the "topic = project" 1:1 mental model — a Slack channel maps better than a Slack thread).

3. **Ack mechanism is not portable.** 👀→👌 reaction-swap is Telegram-native (one-reaction-per-bot constraint). Slack/Discord can stack reactions, so verbatim swap needs an explicit remove+add. LINE has **no reaction API at all** — the abstraction must expose `ackMechanism: "reaction-swap" | "reaction-stack" | "status-line" | "inline-text-fallback"` and the wrapper must pick one at mount time.

4. **Typing/keepalive scope differs.** Telegram: per-target keepalive every 4s (existing `typing-keepalive.ts`). Slack: status auto-clears at 2min, much wider window. Discord: 10s. LINE: only works in 1:1, not groups — must no-op in groups.

5. **Inbound transport polarity is reversed for LINE.** Telegram/Slack/Discord all let the bot **pull** events from a long-running connection the bot opens. LINE **pushes** events into a public endpoint the bot must expose. This breaks the "single-process poller" architecture and forces a second component (webhook receiver + tunnel manager) for LINE specifically. The interface needs a `transport: "pull" | "push"` distinction with separate lifecycle hooks (`startConnection()` vs `startWebhookServer({ publicUrl })`).

6. **File-out byte-vs-URL distinction.** Telegram and Discord accept multipart raw bytes in one call. Slack requires a 3-step dance (get URL → PUT → complete). LINE requires the bot to **host** the file at a public HTTPS URL — the bot can't push bytes. The interface needs `sendFile({ bytes, mime })` for the first three and `sendFile({ publicUrl })` for LINE, or a shared `prepareFile()` helper that the LINE adapter implements by writing to its already-public tunnel.

## Setup friction ranking

Ranked by TTHW for a user with none of the prerequisites:

1. **Telegram (~3 min).** @BotFather → paste token → /pair. No public URL, no scopes, no app review.
2. **Discord (~5 min).** Developer Portal → bot token → enable MESSAGE_CONTENT intent → OAuth invite URL → click-to-add-to-guild. Single-server install is trivial; multi-server needs OAuth distribution UX.
3. **Slack (~5–8 min).** Manifest paste is fast, but **workspace admin install gate** is real in many orgs (could be 5 min or 5 days depending on policy). Socket Mode requires two tokens (app-level + bot), which is a confusion-amplifier.
4. **LINE (~20–30 min).** Provider vs Channel confusion, auto-reply disable in a separate "Official Account Manager" surface, token vs secret confusion, webhook URL + tunnel provisioning, Verify roundtrip. Documented in `line-support.md`.

## Per-platform unknowns

Things not verified during this research that need confirmation before implementation:

- **Slack** — exact behavior when the bot is uninstalled mid-session (Socket Mode disconnect code? auto-reconnect?). Also: does the Enterprise Grid `org-deploy` flow change scopes/install UX materially?
- **Slack** — concrete max edit window for `chat.update` on bot messages when admin policy is "default." Docs are vague; need to test.
- **Slack** — whether `assistant.threads.setStatus` requires the app to be enabled as an "AI app" specifically (the docs hint at a setting). If yes, falling back to reactions is fine; the leaky abstraction already handles this.
- **Discord** — does `MESSAGE_CONTENT` privileged intent require business verification once the bot crosses 100 guilds, or only at 75? Public docs are inconsistent. Not blocking for personal-use install but blocking for any "share with friends" UX.
- **Discord** — bot-created threads in non-forum text channels auto-archive after inactivity (1h / 24h / 3d / 7d configurable). Need to confirm this doesn't break the "topic = project" persistence assumption.
- **Discord** — rate limit per-channel for `POST /typing` when keepaliving for a long-running claude turn (5-15 min). Likely fine, but a 50req/sec global cap × N concurrent topics could bite.
- **LINE** — whether the `messageReactionEvent` exists in any form (search returned nothing definitive). If LINE adds reactions in 2026, the abstraction's `ackMechanism` enum could collapse. Worth re-checking quarterly.
- **LINE** — whether `userId` is stable across token rotation within the same Channel (vs the known per-Channel uniqueness gap). Affects pairing semantics.
- **All four** — outbound rate-limit behavior under the specific pattern this project produces: bursty long messages from one bot to one chat after a long claude turn. Headline numbers say it's fine; haven't measured.
